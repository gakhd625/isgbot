# Appointment Tracking System

Production-ready Next.js 15 appointment tracking system for authenticated appointment setters and passwordless public checker links.

## System Architecture

- Next.js App Router with server components for data loading and client components only for forms, theme controls, and Supabase Realtime subscriptions.
- Supabase Auth protects setter routes through `middleware.ts`.
- Supabase PostgreSQL stores appointments, checker names, and immutable activity logs.
- RLS restricts setters to their own appointments and activity logs.
- Checkers use `/checker/[publicToken]`; access is scoped by UUID public token through `SECURITY DEFINER` RPCs.
- Server Actions validate and sanitize all writes with Zod before calling repositories and services.
- Repository layer isolates database access.
- Service layer handles business workflows, revalidation, audit creation through RPC, and Telegram notifications.
- Supabase Realtime refreshes dashboard counters, lists, activity feed, and checker views.
- CSV export is served by `app/api/export/route.ts`.

## Database Schema

The schema is in `supabase/migrations/202606100001_initial_schema.sql`.

- `appointment_status` enum: `Pending`, `Confirmed`, `Done`, `Reschedule`, `No Show`, `Cancelled`
- `appointments`: client details, setter ownership, status, remarks, checker metadata, timestamps, public UUID token
- `activity_logs`: immutable audit entries for every checker status update
- `checkers`: active checker names

Seed data is in `supabase/seed.sql`: Maria, John, Sarah, Mark.

## Supabase Setup

1. Create a Supabase project.
2. Apply the migrations in `supabase/migrations` in timestamp order.
3. Run `supabase/seed.sql`.
4. Confirm Realtime is enabled for `appointments` and `activity_logs`.
5. Create setter users in Supabase Auth.
6. Copy `.env.example` to `.env.local` and set real values.

## RLS Policies

Policies are in `supabase/migrations/202606100002_rls_policies.sql`.

- Setters can select, insert, and update only rows where `setter_id = auth.uid()`.
- No delete policy exists for appointments.
- Setters can read activity logs only for their own appointments.
- Anonymous checkers can read active checker names.
- Checker appointment reads and updates go through token-scoped RPCs.

## Telegram Checker Board

Appointments are posted to Telegram as one message per appointment with inline status buttons. Checkers tap a button, the webhook updates Supabase, writes the audit log, edits the Telegram appointment message, and refreshes dashboards through Realtime.

Run the latest migrations, then register checker Telegram IDs:

```sql
update public.checkers
set telegram_user_id = 123456789, telegram_username = 'maria_username'
where name = 'Maria';
```

Set the webhook after the app is reachable from the internet:

```txt
https://api.telegram.org/botYOUR_BOT_TOKEN/setWebhook?url=https://YOUR_DOMAIN/api/telegram/webhook&secret_token=YOUR_TELEGRAM_WEBHOOK_SECRET
```

The same `TELEGRAM_WEBHOOK_SECRET` must be set in `.env.local` or Vercel environment variables.

## Workflows

Setter:

1. Log in at `/login`.
2. Create an appointment at `/appointments/new`.
3. Copy the generated checker URL.
4. View/search/filter appointments, dashboard counters, activity history, and reports.
5. Export CSV from `/reports`.

Checker:

1. Open Telegram or `/checker`.
2. View client phone/area/date/time/current status.
3. Tap a Telegram status button or submit an update on `/checker`.
4. The app stores an audit log, timestamps the appointment, edits the Telegram message, and refreshes dashboards through Realtime.

## Reports

The reports screen supports date range, status, and checker filters.

- Daily, weekly, and monthly reports are generated by choosing date ranges.
- Status summary counts appointments by status.
- Checker productivity counts updates by checker from `activity_logs`.
- CSV export supports date range, status, and checker filters.

## Testing Guide

1. Install dependencies: `npm install`.
2. Type-check: `npm run typecheck`.
3. Build: `npm run build`.
4. Run locally: `npm run dev`.
5. Test auth: create a Supabase Auth user, log in, and confirm `/dashboard` is protected.
6. Test appointment creation: create an appointment and verify Telegram receives a message with status buttons.
7. Test checker update: tap a Telegram button or open `/checker`, update status, and confirm:
   - `appointments.status`, `remarks`, `checked_by`, and `checked_at` changed
   - `activity_logs` has a new row
   - Telegram edits the appointment message when credentials and webhook are configured
   - dashboard counters and activity feed refresh
8. Test CSV export from `/reports`.
9. Test RLS from Supabase SQL editor or API client:
   - setter A cannot read setter B rows
   - anonymous users cannot list appointments
   - checker token RPC only returns the appointment for that token

## Vercel Deployment Guide

1. Push the project to a Git provider.
2. Import it in Vercel.
3. Add all environment variables in Vercel Project Settings.
4. Set `NEXT_PUBLIC_SITE_URL` to the production domain.
5. Apply Supabase migrations and seed data before first production use.
6. Deploy.
7. Create setter accounts in Supabase Auth.
8. Send a test appointment through the full setter and checker workflow.
